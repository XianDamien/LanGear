## 3. Backend Review

### 3.1 PRD_MVP_BACKEND.md

**Strengths:**
- Clean 5-layer architecture (Router / Service / Repository / AI Adapter / FSRS Adapter)
- Error codes are well-defined per endpoint
- Time-sequence flows (Section 5) are clear and implementable
- Unified response structure with `request_id` is good practice

**Issues:**

| # | Issue | Severity | Detail |
|---|-------|----------|--------|
| B1 | **Backend directory contains py-fsrs library, not the app** | Critical | The `backend/` directory is literally a clone/copy of the `py-fsrs` package (v6.3.0). There is no FastAPI app, no `app/` directory, no `main.py`, no database models, no routers, no services. The entire backend application layer needs to be built from scratch. |
| B2 | **No ASR integration spec** | High | The core training flow requires: user records audio -> backend transcribes -> AI compares. But there's no specification of which ASR service to use. The old PRD used AssemblyAI. The MVP PRD mentions "转写文本" in the flow but never specifies the ASR provider or API. |
| B3 | **No audio storage implementation spec** | High | Step 2 of the submission flow says "持久化用户音频（若使用对象存储，先上传再写路径）" but doesn't commit to local or OSS storage. This needs a decision. |
| B4 | **AI Adapter has no spec** | High | "AI Adapter Layer" is mentioned in architecture but there's no detail on: which model (Gemini?), what prompt template, what API key management, retry strategy, timeout handling. |
| B5 | **No database migration tooling** | Medium | PRD mentions SQLite but no ORM (SQLAlchemy? Tortoise? raw SQL?), no migration tool (Alembic?). |
| B6 | **Missing CORS configuration** | Medium | Frontend (Vite dev server) and backend (FastAPI) will run on different ports. No CORS spec is mentioned. |
| B7 | **`POST /study/submissions` does too much** | Medium | Single endpoint handles: audio save + ASR transcription + AI feedback generation + FSRS update + DB write. This could be slow (AI call latency). The old architecture solved this with async processing. The MVP mandates synchronous ("no degradation"), which means the user waits for the entire pipeline. Expected latency: 5-15 seconds per card submission. |
| B8 | **No health check endpoint** | Low | Old PRD had `/health`; MVP PRD doesn't mention it. |
| B9 | **py-fsrs package should be a dependency, not vendored** | Medium | The FSRS code is currently copied into `backend/fsrs/`. It should be a pip/uv dependency (`fsrs>=6.3.0`) instead. |

### 3.2 Current Backend Code Analysis

The `backend/` directory is essentially the open-source `py-fsrs` library with its own tests, CI configs, and package metadata. **None of this is the LanGear backend application.** The actual FastAPI app needs to be built from scratch:

- `app/main.py` - FastAPI entry point
- `app/routers/` - API endpoints
- `app/services/` - Business logic
- `app/repositories/` - Database access
- `app/adapters/` - AI + FSRS wrappers
- `app/models/` - SQLAlchemy/Pydantic models
- `app/database.py` - SQLite connection
- `pyproject.toml` - With proper dependencies (fastapi, uvicorn, fsrs, etc.)

## 5. Dictionary Feature Analysis (Your Request)

You mentioned wanting "自主导入词典功能链接词典（可内置也可以导入）". Here's the current situation and options:

### 5.1 Current State
- **MVP PRD**: Explicitly excludes dictionary features. Section 1: "MVP 阶段只关注内置教材训练流程，不包含自定义上传与字幕切分等扩展功能"
- **Archived PRD_vNext.md**: Section 4.4 mentions "词典与 AI 解释：支持内置词典/导入词典"
- **Old PRD.md**: No dictionary feature at all
- **Frontend current code**: No dictionary feature

### 5.2 Feature Scope Options

**Option A: Minimal MVP Dictionary (Inline AI Lookup)**
- During study, user can tap any word in `front_text`/`back_text` to get an AI-generated explanation
- No dictionary database; each lookup is a Gemini/GPT call
- Pros: Zero dictionary data management, works for any language
- Cons: Requires network, costs per lookup, no offline support

**Option B: Built-in Dictionary + Import (Recommended if you want this in MVP)**
- New DB table: `dictionaries` (id, name, type: builtin/imported, language_pair, created_at)
- New DB table: `dictionary_entries` (id, dictionary_id, headword, pronunciation, definitions_json, examples_json)
- Built-in: Ship with ECDICT (open-source EN-CN dictionary, ~770K entries) or CC-CEDICT
- Import: Support uploading StarDict (.dict/.idx/.ifo) or MDDICT (.mdx) format files
- Backend parses and indexes imported dictionaries
- Frontend shows dictionary popup on word tap during study
- Pros: Offline-capable, fast lookup, user can add specialized dictionaries
- Cons: Adds 2 tables, import parsing logic, dictionary UI

**Option C: Dictionary Linking (External)**
- No dictionary data stored locally
- User configures links to external dictionary services (Youdao, Cambridge, etc.)
- Word tap opens external dictionary in iframe/popup
- Pros: Simplest implementation, always up-to-date
- Cons: Requires network, fragmented UX, no control over display

**Option D: Hybrid (AI + Optional Local Dictionary)**
- Default: AI-generated explanations (Option A)
- Optional: User can import local dictionaries that override AI for known words
- Word tap checks local dictionary first, falls back to AI
- Pros: Works without dictionary import, enhanced with it
- Cons: Most complex logic

### 5.3 If Adding Dictionary to MVP, Required Changes

**PRD.md changes:**
- Add dictionary to Section 1 scope
- Add `dictionaries` and `dictionary_entries` tables to Section 6
- Add dictionary-related acceptance criteria to Section 9

**Frontend PRD changes:**
- Add dictionary popup component to StudySessionView
- Add dictionary management page to Settings or a new route
- Add `dictionary` Pinia store
- Add import flow (file upload + parsing progress)

**Backend PRD changes:**
- Add `GET /api/v1/dictionaries` - list dictionaries
- Add `POST /api/v1/dictionaries/import` - upload dictionary file
- Add `GET /api/v1/dictionaries/lookup?word=X` - lookup word
- Add dictionary parsing service (StarDict/MDX/CSV format support)
